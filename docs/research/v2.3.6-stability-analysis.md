# Research Report: Evolution API v2.3.6 Stability Analysis

**Date:** 2025-11-20
**Version Analyzed:** v2.3.6
**Dependency Focus:** `baileys` (v7.0.0-rc.6)

## 1. Executive Summary
This report provides a comprehensive stability analysis of Evolution API v2.3.6 to determine its viability as an upgrade from v2.3.2.

**Conclusion:** **DO NOT UPGRADE to v2.3.6 for production environments.**

While v2.3.6 includes security patches and new features, it introduces severe regressions and depends on an unstable pre-release version of the `baileys` library. The most critical issue is the "zombie instance" or "freeze" bug, where instances become unresponsive, halting all message flow. This, combined with webhook failures and unresolved message sync issues, makes v2.3.6 a high-risk choice that is demonstrably less stable than v2.3.2.

**Recommendation:**
1.  **Remain on v2.3.2.** The known bugs in this version (e.g., PENDING message sync, DELIVERY_ACK failures) are primarily data consistency issues and are less severe than the total operational stoppage caused by the v2.3.6 freeze bug.
2.  **Focus on patching v2.3.2.** Dedicate development efforts to fixing the specific, known issues within the current, more stable codebase.
3.  **Re-evaluate at v2.4.0** or a future stable release that explicitly confirms fixes for the freeze bug and moves `baileys` to a stable, production-ready version.

## 2. Key Findings & Regressions in v2.3.6

Our research, combining GitHub issue analysis and external intelligence, confirms that the stability of v2.3.6 is poor. The problems are not isolated incidents but represent systemic issues.

### Finding 1: The "Freeze" / "Zombie Instance" Bug (Severity: Critical)
This is the most significant regression. Instances appear "Connected" but stop processing messages entirely.
-   **Symptoms:** Webhooks cease firing, API requests to send messages time out, and instances cannot be managed via the API, requiring a full container restart.
-   **Root Cause:** The issue is strongly linked to both the `baileys` v7.0.0-rc.6 dependency and underlying Redis connectivity problems. The Baileys library has known issues with freezing in large, active groups. Additionally, if the Redis cache becomes desynchronized, the API's authentication and instance management logic can deadlock, leading to a freeze.

### Finding 2: Unstable `baileys` Dependency (Severity: Critical)
Evolution v2.3.6 upgraded its core dependency to `baileys` version `v7.0.0-rc.6`. This is a **release candidate**, not a stable production library.
-   **Known `baileys@7.0.0-rc.6` Issues:**
    -   **High disconnection rates:** Frequent, unexplained "device_removed" errors.
    -   **Media sending failures:** A known regression prevents the sending of images, videos, or files.
    -   **Decryption and Sync Errors:** "Bad Mac" errors and incomplete app state syncs are common, leading to message history gaps.
-   **Impact:** By adopting an unstable RC, Evolution v2.3.6 inherits all of its upstream bugs, making it an unsuitable foundation for a production system.

### Finding 3: Persistent Webhook & Message Sync Failures (Severity: High)
The issues you are currently facing with v2.3.2 are not reliably resolved in v2.3.6.
-   **Webhook Failures:** Intermittent webhook loss remains a significant problem, with community reports indicating up to 50% of events can be lost. This is sometimes tied to DNS instability but points to a brittle webhook dispatching mechanism.
-   **PENDING Message Sync:** The root cause, an incomplete WhatsApp message structure being saved to the database, triggers `PrismaClientValidationError`. This issue persists in v2.3.6, as the core logic for handling these special message types has not been fixed.
-   **DELIVERY_ACK & `interactiveMessage`:** Sync failures for group media (`DELIVERY_ACK`) and ad-related greetings (`interactiveMessage`) are not addressed in v2.3.6. The core message processing pipeline still lacks support for these specific message subtypes.

### Finding 4: Regressions from Intermediate Versions (Severity: High)
The development path from v2.3.2 to v2.3.6 was turbulent. Versions v2.3.3 and v2.3.4 introduced major bugs that were not fully resolved by v2.3.6.
-   **Known Regressions:**
    -   **Reply/Quote functionality breaks.**
    -   **Message Mixing:** A critical bug where messages were incorrectly attributed to the wrong users.
    -   **Data Corruption:** An issue where the API incorrectly added a digit to Brazilian phone numbers.
-   **Community Consensus:** Multiple reports explicitly state that v2.3.2 is more stable than v2.3.3 and subsequent versions, reinforcing the decision to avoid the upgrade.

## 4. Final Recommendation
The research overwhelmingly indicates that **v2.3.6 is not a stable release**. The introduction of an unstable `baileys` release candidate is the primary driver of new, critical issues that are more severe than the known data-consistency bugs in v2.3.2.

**Actionable Strategy:**
1.  **Halt all plans to upgrade to v2.3.6.**
2.  **Mitigate the security risk in v2.3.2** by applying network-level controls (e.g., firewall rules) to restrict access to the vulnerable `/assets` endpoint.
3.  **Create targeted fixes for the known bugs in the v2.3.2 codebase.** This provides the most direct path to improving stability without introducing new, unknown risks.
4.  **Monitor upstream `evolution-api` and `baileys` projects.** Do not consider another upgrade until both projects have released new, designated *stable* versions and community feedback confirms the resolution of the freeze bug.
