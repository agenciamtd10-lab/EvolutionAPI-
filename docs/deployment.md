# Deployment Guide

## Overview
We use a custom `build.sh` script to build and push the Docker image to Azure Container Registry (ACR). The script automatically generates a tag based on the version and git commit hash.

## Prerequisites
- **Azure CLI**: Installed and logged in (`az login`).
- **Docker**: Installed and running.
- **Access**: Permission to push to `prospek.azurecr.io`.
- **GitHub Codespaces**: If building from Codespaces, ensure you have authenticated with Azure (`az login --use-device-code`).

## Image Tagging Strategy
We follow a standardized naming convention for image tags:
`v<version>-<project>-<commit_hash>`

Example: `v2.3.2-evolution-b685478`
- **Version**: Taken from `package.json` (e.g., `2.3.2`).
- **Project**: `evolution`.
- **Commit**: Short Git commit hash (e.g., `b685478`).

## Dependency Management

### Upstream Alignment
This fork stays as close as possible to the upstream Evolution API. However, one critical deviation exists for build stability:

**Baileys Dependency:**
- **Upstream v2.3.2**: Uses `"baileys": "github:WhiskeySockets/Baileys"` (requires source compilation)
- **Our v2.3.2**: Uses `"baileys": "6.7.19"` (npm registry, pre-built)
- **Reason**: The GitHub version requires compiling libsignal-node during `npm install`, which fails on modern systems due to breaking changes since the original v2.3.2 release
- **Compatibility**: Version 6.7.19 is functionally equivalent to upstream's intended version

**General Principle**: When upstream adds dependencies from GitHub sources, check if pre-built npm versions exist to ensure builds remain reproducible over time.

## Pre-Build Checklist

Before running `./build.sh`, verify:

- [ ] **Branch**: On `production-v2.3.2` (`git branch --show-current`)
- [ ] **Clean state**: No uncommitted changes (`git status`)
- [ ] **Dependencies verified**: Test build succeeds in Codespaces
  ```bash
  npm ci && npm run build
  ```
- [ ] **Azure authenticated**: `az login --use-device-code` completed

If `npm ci` or `npm run build` fails, see troubleshooting section below before proceeding with Docker build.

## Deployment Steps

### 1. Prepare the Environment (GitHub Codespaces)
1.  **Create Codespace**: Create a new codespace from the **`production-v2.3.2`** branch.
2.  **Login to Azure**:
    ```bash
    az login --use-device-code
    ```
    Follow the on-screen instructions to authenticate via your browser.

### 2. Verify Branch
Ensure you are on the correct branch:
```bash
git branch --show-current
# Output should be: production-v2.3.2
```

### 3. Run Build & Push
Execute the build script:
```bash
./build.sh
```
This script will:
1.  Read the version from `package.json`.
2.  Get the current git commit hash.
3.  Build the Docker image.
4.  Tag it (e.g., `v2.3.2-evolution-b685478`).
5.  Push it to `prospek.azurecr.io/evolution:<tag>`.

### 4. Update Azure Container App
1.  Log in to the [Azure Portal](https://portal.azure.com).
2.  Navigate to **Container Apps** > **[Your Evolution App Name]**.
3.  Go to **Application** > **Containers**.
4.  Click **Edit and Deploy** (or "Create new revision").
5.  Select the container image and update the **Image tag** to the one generated by the script (e.g., `v2.3.2-evolution-b685478`).
6.  Click **Save** / **Create** to deploy the new revision.

## Troubleshooting
- **Login Failed**: Run `az login --use-device-code` again.
- **Permission Denied**: Ensure your Azure user has `AcrPush` role on the `prospek` registry.
- **Dirty State**: If you have uncommitted changes, the build will still run, but the commit hash in the tag might be misleading. Ideally, commit your changes before building.
- **npm install/ci fails during dependency build**:
  - **Symptom**: Errors like "git dep preparation failed" or TypeScript compilation errors from baileys/libsignal-node
  - **Cause**: Git-sourced dependencies may fail to compile due to breaking changes in their transitive dependencies
  - **Solution**:
    1. Check if the dependency has a pre-built npm registry version
    2. Update `package.json` to use the registry version (e.g., `"baileys": "6.7.19"` instead of `"baileys": "github:WhiskeySockets/Baileys"`)
    3. In Codespaces: `rm -rf node_modules package-lock.json && npm install`
    4. Regenerate `package-lock.json` in Codespaces and commit it
  - **Note**: Our v2.3.2 uses `baileys@6.7.19` from npm registry (differs from upstream) to ensure reproducible builds. See git history for context.
